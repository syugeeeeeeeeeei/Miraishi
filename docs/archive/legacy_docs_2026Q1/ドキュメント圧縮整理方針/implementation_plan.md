# 実装計画

## 方針（推奨）
- `docs/readmes` は引き続き正本（要件・仕様・用語・操作）として維持する。
- `readmes` 以外の履歴ドキュメントは「検索しやすい要約」と「必要時に辿れる原本」の2層に再編する。
- 今後は「1修正=3ファイル」を廃止し、「1修正=1エントリ（追記型）」へ統一する。

## 新しい構成案

```text
docs/
  readmes/                      # 正本（現状維持）
  history/
    README.md                   # 履歴ドキュメントの読み方
    index.md                    # 全修正の索引（年月日・カテゴリ・PR/コミット等）
    by_domain/
      yaml_editor.md            # 系列ごとの統合サマリー
      pdf_export.md
      tax_schema.md
      scenario_ui.md
      misc.md
    timeline.md                 # 時系列の主要変更ログ（1行要約）
  archive/
    legacy_docs_2026Q1/         # 旧63トピック（必要なら退避）
```

## 圧縮ルール

1. 3点セット統合
- 各トピックの `task / implementation_plan / walkthrough` から以下のみ抽出して1レコード化する。
  - 目的（何を直したか）
  - 主要変更（何が変わったか）
  - 影響範囲（主要ファイル）
  - 検証結果（成功/未実施）
  - 備考（仕様策定のみ等）

2. 系列統合
- 近いテーマを1ファイルへ集約する（例: YAMLエディター群、比較PDF群、税制スキーマ群）。
- 同系統内では時系列で並べ、再発防止や設計判断の変遷が追える形にする。

3. 索引の一本化
- `history/index.md` に全修正を1行単位で登録する。
- 例: `2026-03-27 | tax_schema | 税制スキーマ_文字式DSL対応 | summaryリンク`

4. 原本の扱い
- すぐ削除しない。まず `docs/archive/legacy_docs_2026Q1` に移動。
- 1〜2スプリント運用して問題なければ、タグを打ってから削除または別ブランチ保管。

## 段階移行ステップ

1. `history` 骨組み作成
- `README.md / index.md / timeline.md / by_domain/*.md` を作成。

2. 63トピックを機械的に正規化
- タイトル・日付・主要変更・検証結果を抽出し、`index.md` と `by_domain` に追記。

3. 旧フォルダ退避
- 既存63トピックを `archive/legacy_docs_2026Q1` へ移動。

4. 最終確認
- `docs/readmes` から履歴導線リンクを追加。
- 開発者が「現状仕様」と「過去判断」を3クリック以内で辿れることを確認。

## 今後の運用ルール（増殖防止）
- 新規修正時は `history/index.md` と該当 `by_domain/*.md` への追記のみ。
- 例外的に詳細設計が必要な大規模機能だけ、個別フォルダを一時作成し、完了後に `history` へ統合して閉じる。
- 月次で1回、`history/timeline.md` を更新して重要変更を再要約する。

## 期待効果
- 参照対象を `63フォルダ/189ファイル` から、日常運用では `history` 配下の少数ファイルへ圧縮できる。
- 「読む順番」「探す場所」「残す粒度」が明確になり、メンテコストを大幅に下げられる。
