# 仕様書（互換契約版）

## 1. 文書の目的

本書は、Miraishi の実装方式ではなく「システム契約」を定義する。  
内部構成やUIが変化しても、ここで定義した契約が守られる限り仕様準拠とする。

## 2. システム構成（論理）

Miraishi は以下3層の責務分離を前提とする。

- ドメイン・実行層（計算、永続化、OS連携）
- ブリッジ層（安全な呼び出し窓口）
- 表示・操作層（入力、可視化、状態管理）

### 2.1 層ごとの責務

- ドメイン・実行層
  - シナリオの永続化
  - 予測計算
  - 税制スキーマ読み込み
- ブリッジ層
  - 表示層へ公開するAPI契約の固定
  - 不要なOS能力の隠蔽
- 表示・操作層
  - ユーザー入力
  - 複数シナリオ選択
  - 表/グラフ表示

## 3. データ契約

本節は「最低限維持されるべき項目」を定義する。  
追加項目は許可される（後方互換必須）。

### 3.1 シナリオ契約（Scenario）

必須コア項目:

- `id`: シナリオ一意識別子
- `title`: 表示名
- `initialBasicSalary`: 初期基本給（月額）
- `annualBonus`: 年間ボーナス
- `salaryGrowthRate`: 年間成長率
- `allowances`: 手当配列
- `overtime`: 残業条件
- `probation`: 試用期間条件
- `deductions`: 控除条件
- `createdAt` / `updatedAt`

互換ルール:

- 既存コア項目の意味変更は禁止。
- 追加項目は任意（省略可能）として導入する。
- 未知項目は読み込み時に無視可能であること。

### 3.2 グラフ設定契約（GraphViewSettings）

必須コア項目:

- `predictionPeriod`: 予測年数
- `averageOvertimeHours`: 月平均残業時間
- `displayItem`: 表示対象

互換ルール:

- 既存表示項目の意味は変更しない。
- 新規表示項目は追加のみ許可（既存利用を破壊しない）。

### 3.3 予測結果契約（PredictionResult）

年次明細配列を返す。各年次は少なくとも以下を持つ。

- `year`
- `grossAnnualIncome`
- `netAnnualIncome`
- `totalDeductions`
- `breakdown.income.*`
- `breakdown.deductions.*`

## 4. 処理仕様

## 4.1 シナリオ保存

- 入力データは保存前に妥当性検証を行う。
- 永続化更新は論理的に「読込→変更→保存」で一貫処理する。

## 4.2 予測計算

計算は次の概念順で行う。

1. 予測期間ループ（年次）
2. 収入側の算定（基本給、賞与、手当、残業）
3. 社会保険料の算定
4. 所得控除の算定
5. 課税所得の算定
6. 税額算定（所得税・住民税）
7. 手取りと内訳の確定

算出規則:

- 入力単位は通貨（円）と時間（時間/月）を基本とする。
- 表示用数値は円単位で丸める。
- 同一入力・同一税制スキーマでは同一出力を返す。

## 4.3 表示更新

- 入力変更または表示設定変更時に、必要な再計算を実施する。
- 複数シナリオは同一設定条件で比較される。

## 5. 永続化仕様

- シナリオ集合をローカルストレージへ保存する。
- 保存形式は将来拡張を許容する構造化データとする。
- 既存データを失わない移行を前提にする。

## 6. 税制スキーマ仕様

- 税率・保険料率・控除額は外部スキーマから供給する。
- 計算ロジック本体と税制データは分離する。
- スキーマ更新で法改正追従を可能にする。

## 7. エラー契約

- 保存失敗・計算失敗時は、呼び出し元へ失敗情報を返却する。
- 失敗時でもアプリ全体の操作継続性を優先する。
- 起動必須リソース欠損時は安全に停止できること。

## 8. セキュリティ契約

- 表示層はブリッジ層を通じてのみ特権操作を要求できる。
- 外部リンク遷移は安全な方法で制御する。
- 機密情報は設定ファイル・環境変数として管理し、UIへ露出しない。

## 9. 性能契約

- 体感遅延を抑えるため、再計算コストは入力規模に応じて管理する。
- キャッシュ最適化は許容するが、整合性を優先する。

## 10. テスト契約

最小保証:

- 型整合性検証（typecheck）
- 計算ロジックのユニットテスト
- ビルド成立確認

推奨:

- 主要ユーザーフローの統合/E2E検証
- 異常系（不正入力、読み込み失敗）の検証

## 11. 互換性・拡張ルール（重要）

本書を将来改訂不要にするため、実装変更は次を順守する。

- コア契約の「削除・意味変更」は行わない。
- 変更は「追加」「任意化」「内部最適化」を優先する。
- 新機能は既存フローに対して後方互換で組み込む。
- UI改修は操作目的を維持し、到達性を損なわない。

このルールが守られる限り、本仕様書は継続有効である。
